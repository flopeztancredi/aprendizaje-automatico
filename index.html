01

Fundamentos y ciclo de Machine Learning

Qué es el aprendizaje automático

El aprendizaje automático, o Machine Learning, es un subcampo de la Inteligencia Artificial que permite construir modelos capaces de aprender patrones a partir de datos y utilizarlos para predecir, clasificar, agrupar, recomendar o detectar anomalías en casos nuevos. En lugar de programar todas las reglas posibles, se entregan ejemplos a un algoritmo para que ajuste sus parámetros, los valores internos donde queda representado lo aprendido. El resultado no piensa ni comprende: aplica patrones aprendidos.

La Inteligencia Artificial es el campo más amplio e incluye también enfoques que no aprenden de datos, como sistemas basados en reglas, búsqueda y planificación. Deep Learning es, a su vez, una rama de Machine Learning que utiliza redes neuronales, modelos formados por capas de unidades conectadas, con varias capas internas.

Qué puede aprender y dónde se limita

Machine Learning mira el pasado, abstrae patrones y los aplica a casos nuevos. Es potente cuando la tarea está tipificada, es decir, cuando el procedimiento está bien conocido y la variabilidad posible es acotada. Si aparece una situación completamente nueva, fuera de los patrones observados, el modelo no tiene experiencia desde la cual extrapolar.

La IA actual no es inteligencia humana, sino un mecanismo estadístico o probabilístico que produce resultados convincentes. Puede extrapolar dentro de un dominio aprendido, pero no transfiere conocimiento entre dominios con la flexibilidad de una persona.

El robot que dobla una camisa muestra esta limitación. Para una persona es una tarea cotidiana, pero cada camisa cae de una forma distinta y genera muchísimas configuraciones posibles. La dificultad no es solo la motricidad fina, sino resolver una situación variable usando abstracción y conocimiento general. El mejor resultado surge al complementar IA con inteligencia natural: la máquina aporta velocidad y repetición, la persona aporta criterio, abstracción y conocimiento del problema.

Las piezas mínimas

  • Dataset: conjunto de datos usado para desarrollar el sistema. Cada fila suele representar una observación, como una venta, un cliente o una medición.
  • Feature o característica: variable de entrada que el modelo utiliza, como precio, temperatura, fecha o cantidad comprada.
  • Target: resultado que se quiere predecir cuando existe una respuesta conocida, como fraude o no fraude, ventas futuras o tiempo hasta una falla.
  • Algoritmo de aprendizaje: procedimiento que busca una relación entre las entradas y el resultado, o una estructura dentro de los datos.
  • Entrenamiento: proceso de ajustar el algoritmo con datos históricos.
  • Modelo: resultado obtenido después del entrenamiento, con los valores y relaciones ya aprendidos.
  • Inferencia: uso del modelo entrenado sobre datos nuevos para obtener una predicción o decisión.
  • Métrica: número que mide un aspecto del resultado, como la proporción de aciertos o el tamaño promedio del error.
  • Validación: evaluación del modelo con datos que no utilizó para ajustar sus parámetros.

En un e-commerce, una fila puede ser una compra, las features pueden incluir precio, fecha, producto y cliente, y el target puede indicar si habrá devolución. El entrenamiento usa compras pasadas y la inferencia estima qué ocurrirá con una compra nueva.

ML no se construye igual que el software tradicional

La diferencia sustancial frente al software tradicional es el sentido del flujo de datos. Una aplicación tradicional suele ser productora de datos: su operación, transacciones y registros de actividad generan información. Un sistema de ML es primero un gran ingestor de datos: necesita historia para entrenarse y abstraer el modelo. En producción puede generar una salida muy pequeña, por ejemplo una clase o un número, en comparación con el volumen usado para desarrollarlo.

El ciclo de vida

  1. Validar el problema y definir el criterio de éxito.
  2. Investigar viabilidad y acceso a los datos.
  3. Realizar el análisis exploratorio de datos (EDA, del inglés Exploratory Data Analysis) y decidir Go (continuar) o No Go (detener el proyecto).
  4. Limpiar los datos y hacer ingeniería de características (transformar o crear variables para que el modelo pueda representar mejor el problema).
  5. Entrenar, validar y ajustar el modelo.
  6. Explicar resultados y detectar sesgos.
  7. Diseñar la interfaz y planificar el uso productivo.
  8. Desplegar el modelo registrando su versión, cómo reproducirlo y cómo observar su funcionamiento.
  9. Monitorear desempeño y cambios en los datos o en su comportamiento.
  10. Reentrenar o iniciar una nueva iteración cuando corresponda.
  11. Retirar el sistema de forma controlada y documentar el cierre.

La gestión del conocimiento y la trazabilidad atraviesan todas las etapas.

Por qué no es lineal

El proyecto avanza con información incompleta. El EDA puede revelar que los datos no alcanzan. La validación puede mostrar patrones heterogéneos. La explicabilidad puede revelar una variable irrelevante o sesgada. Producción puede mostrar que cambiaron los datos o su relación con el resultado. Cada hallazgo puede obligar a volver a una etapa anterior.

La ciencia de datos trabaja con un grado alto de incertidumbre. Por eso requiere más método, no menos: planificación, validaciones, puntos de Go o No Go, bitácora y posibilidad de reprocesar. Un dataset termina de entenderse cuando el modelo ya está funcionando, porque recién entonces aparecen errores, patrones y comportamientos que no eran visibles al comienzo.

Cada etapa debe tener un entregable, una versión identificable y un criterio de Go o No Go.

  • Go: el problema merece ML, hay datos accesibles y representativos, recursos suficientes, una métrica de éxito acordada y riesgos manejables.
  • No Go: el problema se resuelve mejor con reglas, faltan datos críticos, la calidad es irrecuperable, no hay acceso, el costo es inviable o el riesgo supera el valor.
El estudio de viabilidad termina después del EDA. Puede consumir entre 25% y 33% del esfuerzo total. Con sus resultados se reestiman preparación, entrenamiento y producción.

Gestión del conocimiento

Cada extracción, error, transformación, métrica, decisión y resultado debe quedar en una bitácora y alimentar una base de conocimiento de proyectos. La experiencia externa ayuda, pero no reemplaza la experiencia propia de la empresa: sus sistemas, personas, permisos, tiempos y formas de gestión tienen una idiosincrasia que no se transfiere de manera directa desde otra organización.

La figura del científico de datos itinerante, que salta de industria en industria aplicando la misma receta, pierde valor a medida que los problemas se vuelven más específicos. Respuesta: un proyecto de ML es un ciclo iterativo que empieza validando si el problema y los datos justifican ML, continúa con preparación, entrenamiento y controles éticos, y termina con monitoreo, retiro controlado y documentación. El modelo es solo uno de sus componentes.

02

Definición del problema y viabilidad

Definir el problema no es ponerle un título. Es describir el proceso, la decisión que se quiere mejorar y el resultado esperado hasta que negocio y equipo técnico estén hablando de lo mismo. Esa descripción permite contestar dos preguntas distintas: ¿se puede resolver? y ¿podemos resolverlo nosotros con los datos, el tiempo y los recursos disponibles?

Las preguntas iniciales

  1. ¿El problema amerita Machine Learning? Si una regla fija, una consulta o una automatización convencional lo resuelve, ML agrega costo y riesgo sin aportar valor.
  2. ¿Ya fue resuelto? Revisar artículos, estudios, casos, algoritmos, datasets y métricas usadas evita repetir trabajo y permite estimar viabilidad.
  3. ¿Qué se quiere predecir o decidir? Definir target, unidad de análisis, horizonte temporal y usuario de la salida.
  4. ¿Cómo se medirá el éxito? Acordar antes del entrenamiento una métrica técnica y el valor mínimo aceptable para el negocio.
  5. ¿Existen los datos y se puede acceder? Existencia y acceso son problemas distintos.
  6. ¿Hay capacidad? Datos, cómputo, almacenamiento, conocimiento del dominio, seguridad, tiempo y presupuesto.
Sesgo que se evita. Si se elige la métrica después de ver los resultados, se puede favorecer al algoritmo que mejor luce, no al que resuelve el problema. El criterio de éxito se fija primero.

Éxito técnico y éxito de negocio

El criterio de éxito debe incluir una métrica técnica, un valor mínimo aceptable, el costo de cada error, tiempo de respuesta, costo operativo y utilidad para quien usa la predicción. En un modelo médico que clasifica pacientes como enfermos o sanos, un falso negativo es un paciente enfermo clasificado como sano. Cuando ese error es grave, el recall, proporción de enfermos detectados, puede ser más importante que la accuracy, proporción total de aciertos.

  • Detección de cáncer: si 95% de los pacientes están sanos, responder siempre “sano” produce 95% de accuracy y 0% de recall sobre los enfermos. El objetivo no es acertar la mayoría total, sino detectar la mayor proporción de enfermos, aun aceptando más falsos positivos. Por eso importa el recall.
  • ML innecesario: si una regla o consulta alcanza el criterio de éxito con menor costo y riesgo, la decisión correcta es No Go para ML.

Estimación en dos tramos

No conviene prometer todo el proyecto cuando todavía se desconoce la calidad real. Primero se estima hasta el EDA. Con el informe de datos se reestiman limpieza, features, entrenamiento, infraestructura y producción. Este es el llamado doble proyecto.

En un proyecto petrolero, después de diez días la primera presentación alcanzó aproximadamente la mitad de lo esperado inicialmente. No fue una anomalía del proyecto: mostró cuánto cambia la estimación cuando se empieza a conocer el problema y los datos reales. La planificación debe admitir esa incertidumbre y revisar alcance, tiempos y viabilidad con evidencia.

Respuesta: la viabilidad tiene dos niveles: pertinencia de ML para el problema y capacidad de la organización para implementarlo. La decisión Go o No Go se toma después del EDA, con acceso, calidad, recursos, riesgos y criterio de éxito ya evaluados.

03

Datos, acceso y análisis exploratorio

Existencia, acceso y capacidad

Que una empresa afirme que los datos existen no significa que el equipo pueda usarlos. Pueden faltar permisos, anonimización, cumplimiento normativo, acuerdos entre áreas, claves de unión o infraestructura. Solo con acceso efectivo se conocen volumen, nivel de detalle o granularidad, formato y calidad.

Estimación inicial de almacenamiento. Reservar aproximadamente 2,25 veces el tamaño del dataset original permite trabajar sobre una copia, sin consultar ni modificar continuamente el sistema transaccional, y conservar estados intermedios. Con una gestión más agresiva puede bajarse a 1,75, eliminando o archivando esos estados, pero se pierde la posibilidad de reprocesar inmediatamente desde cualquier paso. Es una heurística empírica, no una constante universal.

El análisis exploratorio de datos (EDA, del inglés Exploratory Data Analysis) es la inspección sistemática de los datos antes de entrenar: busca entender su estructura, calidad, distribución y anomalías sin modificar todavía el original.

Qué hace el EDA

  • Estructura: filas, columnas, target, claves, tipos, memoria, granularidad y períodos cubiertos.
  • Calidad: nulos, duplicados, rangos imposibles, categorías inconsistentes, codificación y errores de captura.
  • Comportamiento: distribuciones, valores únicos, desbalance, correlaciones, estacionalidad, cambios y outliers.

Secuencia de trabajo

  1. Preservar una copia inmutable de los datos originales.
  2. Entender cantidad de filas y columnas, esquema, tipos y uso de memoria.
  3. Optimizar tipos si mejora rendimiento, sin alterar el significado.
  4. Revisar valores únicos, rangos y frecuencias por columna.
  5. Cuantificar nulos y duplicados, también por segmentos.
  6. Visualizar distribuciones y relaciones.
  7. Investigar anomalías con consultas dirigidas y con expertos.
  8. Documentar cada hallazgo, hipótesis, evidencia y pregunta abierta.
  9. Emitir informe de calidad y recomendación Go o No Go.

Optimización de tipos: una columna guardada como número decimal, tipo float, que solo contiene enteros puede convertirse a un tipo entero. Si un entero de 64 bits, int64, solo almacena valores entre 1 y 100, puede usarse un tipo menor. Antes de convertir hay que revisar rango, precisión, nulos y valores futuros posibles.

Arqueología de datos. El EDA es un movimiento de ida y vuelta: observar una letra extraña en un código, filtrar registros, comparar facturas, preguntar al área y reconstruir el proceso que produjo el dato. La intuición genera hipótesis, no conclusiones.

Nulos y duplicados

No existe un porcentaje mágico aplicable a todos los proyectos. Un 1% de nulos en millones de filas puede tolerarse, mientras 5% a 7% en pocos miles puede ser grave. La decisión correcta depende de cantidad absoluta, patrón de ausencia, variable afectada, sesgo introducido y posibilidad de imputación.

SituaciónPregunta correctaPosibles acciones
Nulo¿Ausencia real, error de captura o dato no aplicable?Eliminar la fila, completar el valor mediante una estimación, crear un indicador o recuperar la fuente.
Duplicado¿Repetición inválida o varios eventos legítimos iguales?Eliminar, consolidar registros repetidos según una clave o mantener.
Categoría inconsistente¿Mayúsculas, espacios o códigos distintos expresan lo mismo?Normalizar con reglas trazables y validar con negocio.
Rango imposible¿Error técnico, ajuste operativo o fraude?Investigar origen antes de corregir.

Outlier no significa error

Un outlier es una observación fuera de norma. Puede ser un error, un fraude, un evento extraordinario o una transacción válida pero rara. Detectarlo corresponde al EDA. Decidir si se elimina, transforma, segmenta o modela aparte corresponde a la preparación y depende del objetivo y del algoritmo.

  • Moldavsky y las camperas: un comerciante de Tierra del Fuego compró aproximadamente 35 camperas en un local de Once para revenderlas. La operación fue válida y beneficiosa, pero no representaba la venta cotidiana. Es una anomalía, no un error. Si se deja sin análisis, puede forzar al modelo a explicar como habitual un caso ocurrido una sola vez en diez años.
  • Black Friday: las ventas masivas son correctas, pero no representan el funcionamiento habitual, llamado Business as Usual. Pueden separarse y alimentar otro modelo con features de promociones, descuentos y eventos.
  • Aseguradora: la prueba “ingresar menos uno y guardar” confirmó que una aplicación aceptaba pagos negativos. Una preimpresión semanal permitía calcular contramovimientos para forzar balances. El análisis reconstruyó en 20 días una práctica que llevaba entre cuatro y cinco años y había contaminado información enviada al organismo regulador.
  • E-commerce: las notas de crédito deben vincularse con la venta original. En el caso visto, compras al inicio del mes y devoluciones al precio actualizado producían una ganancia por inflación y contaminaban las estadísticas de ventas.
  • SUBE y serendipia: las tarjetas defectuosas se concentraban en una posición de cada caja. El patrón permitió descubrir que el apilado dañaba los chips antes de la grabación. Fue un objetivo secundario o hallazgo por serendipia: conocimiento útil que apareció mientras se investigaba otra cosa.

En una aseguradora, el ingeniero de datos puede ser la única persona que revise los registros uno por uno. Ese análisis funciona también como ciberseguridad reactiva, porque descubre después de la grabación lo que los controles en tiempo real no vieron. Además, comunicar un fraude o un error estructural exige asertividad: hay que sostener el hallazgo con evidencia, reconstruir su historia y explicarlo a quienes deben decidir.

La experiencia de EDA se acumula en tres niveles: problemas generales como nulos y tipos incorrectos, patrones propios de una industria y particularidades de los sistemas de cada empresa. OSDU, Open Subsurface Data Universe, es una referencia para organizar datos de petróleo y gas. Para “afinar el ojo” hay que practicar con datasets distintos, por ejemplo los disponibles en Kaggle, y registrar qué hipótesis funcionaron.

Herramientas y formatos

Pandas, una biblioteca de Python para trabajar con tablas, permite la inspección fina. Herramientas de EDA automático como DataPrep o AutoViz generan perfiles, mapas de calor y gráficos de distribución con poco código. Automatizan cálculos, no la interpretación. Para guardar resultados intermedios, CSV conserva una tabla como texto legible. Joblib puede guardar objetos de Python en formato binario, pero requiere un entorno compatible para volver a abrirlos.

Respuesta: el EDA describe estructura, calidad y comportamiento sin modificar el dato original. Revisa nulos, duplicados, inconsistencias, distribuciones, correlaciones y outliers. El informe resultante define la limpieza y aporta evidencia para el Go o No Go.

04

Limpieza e ingeniería de características

La ingeniería de características, también llamada feature engineering, consiste en transformar las variables disponibles o crear nuevas para que los patrones relevantes queden representados de una forma que el algoritmo pueda usar. Transformar no significa alterar el significado del dato: significa cambiar su representación para hacer visibles regularidades útiles.

Tres conceptos que no deben mezclarse

  • EDA, diagnosticar: observar, cuantificar, formular hipótesis y documentar. No corregir todavía.
  • Data cleaning, sanear: corregir o tratar nulos, duplicados, formatos, categorías y errores para obtener calidad mínima.
  • Feature engineering, representar: crear variables que expongan patrones relevantes para el algoritmo.

Ejemplos de ingeniería de características

  • Descomponer fecha y hora en año, mes, día, hora, día de semana, fin de semana, estación y trimestre. El timestamp original normalmente se excluye del entrenamiento si ya no aporta un patrón útil.
  • Codificar categorías de forma compatible con la familia de modelos.
  • Reescalar magnitudes para evitar que diferencias de escala dominen el aprendizaje.
  • Aplicar transformaciones logarítmicas a distribuciones muy asimétricas cuando tenga sentido.
  • Construir ratios, acumulados, ventanas temporales o interacciones basadas en conocimiento del dominio.
  • Eliminar redundancias o variables sin utilidad comprobada.
  • Reducir varias variables a componentes mediante análisis de componentes principales, PCA, cuando se acepta perder interpretabilidad directa.

PCA, análisis de componentes principales, combina variables correlacionadas en un número menor de componentes numéricos. Puede reducir dimensiones y cómputo, pero cada componente deja de representar una variable de negocio directa.

No saturar al algoritmo. Más columnas no significa más conocimiento. Features irrelevantes agregan almacenamiento, tiempo, riesgo de ruido y complejidad. La selección debe comparar desempeño, estabilidad, costo e interpretabilidad.

Selección y reducción

Antes de entrenar puede estimarse relevancia revisando cuánto varía una columna, cuántos valores distintos contiene, cómo se relaciona con otras variables y qué aporta según el conocimiento del problema. La importancia real se contrasta luego del entrenamiento. Si varias features tienen peso casi nulo y al retirarlas la métrica pasa, por ejemplo, de 0,82 a 0,81, la simplificación puede ser conveniente.

Ejemplo de costo: si cinco features ya alcanzan 85%, un modelo con decenas de variables que tarda tres horas para sostener ese 85% puede ser peor decisión que otro con alrededor de diez variables, 81% y diez minutos de entrenamiento. La diferencia se multiplica en cada validación y reentrenamiento. También puede ser útil retirar correlaciones obvias para que el análisis revele relaciones menos evidentes, siempre verificando que no se elimine información necesaria.

Si las métricas quedan persistentemente muy por debajo de lo esperado, una hipótesis es que faltan features que describan el patrón. No es una demostración automática: también puede fallar el target, la calidad, el algoritmo, la separación de datos o la propia posibilidad predictiva.

Responsabilidades superpuestas

El ingeniero de datos realiza la preparación genérica. El científico de datos puede revisarla al elegir la familia de algoritmos, porque el tratamiento de outliers, el escalado y la reducción dependen del modelo.

Leakage o fuga de información: ocurre cuando el entrenamiento recibe un dato que no estaría disponible al momento de predecir, o cuando información del conjunto de prueba, reservado para la evaluación final, se filtra hacia el entrenamiento. La métrica parece mejorar, pero no representa el uso real.

Respuesta: la ingeniería de características cambia la representación del dataset, por ejemplo descomponiendo fechas o creando ratios. Usa técnicas estadísticas, conocimiento del dominio y experiencia. Una feature se conserva si mejora desempeño, estabilidad o interpretabilidad sin introducir leakage.

05

Entrenamiento, validación y métricas

Las tres familias de problemas

En el aprendizaje supervisado, cada ejemplo incluye el target correcto que el modelo debe aprender a predecir. En el aprendizaje no supervisado, no existe ese resultado conocido y se buscan estructuras dentro de los datos.

Clasificación, regresión y clustering. Las dos primeras son familias de problemas supervisados y la tercera es no supervisada.

  • Clasificación: aprendizaje supervisado que predice una categoría a partir de ejemplos con target, como fraude o no fraude.
  • Regresión: aprendizaje supervisado que predice un valor continuo, como demanda, precio o tiempo hasta falla.
  • Clustering: aprendizaje no supervisado que busca grupos o estructura sin una etiqueta objetivo conocida.
Taxonomía alternativa. También se habla de aprendizaje supervisado, no supervisado y por refuerzo. En aprendizaje por refuerzo, un programa que toma acciones, llamado agente, interactúa con un entorno y aprende a partir de recompensas o penalizaciones. Ante una pregunta por tres familias, conviene responder primero clasificación, regresión y clustering, y luego aclarar cómo se agrupan por paradigma.

Por qué se habla de familias y no de un mejor algoritmo

Los patrones y su distribución interna son invisibles antes de entrenar. Dos empresas pueden querer predecir las mismas ventas y tener columnas con los mismos nombres, pero distinto volumen, historia, concentración y distribución de casos. Por eso un algoritmo puede funcionar mejor en una y otro algoritmo de la misma familia en la otra.

La selección práctica consiste en identificar primero la familia adecuada al tipo de salida y luego comparar varios algoritmos de esa familia con la métrica definida para el problema. También pueden aparecer los nombres caracterización o clustering, clasificación y predicción o regresión.

Mapa mínimo de algoritmos

La familia define qué tipo de salida se necesita. El algoritmo es el procedimiento concreto que se prueba dentro de esa familia. No hace falta memorizar su implementación, pero sí reconocer qué problema resuelve y qué exige de los datos.

  • Regresión lineal: predice un número mediante una relación lineal. Es rápida y sus coeficientes son fáciles de interpretar.
  • Regresión logística: pese al nombre, clasifica. Estima la probabilidad de una clase, por ejemplo spam o no spam.
  • Árbol de decisión: aprende reglas del tipo si ocurre esto, entonces aquella salida. Es explicable, pero un árbol muy profundo puede sobreajustarse.
  • Random Forest: combina muchos árboles y decide por voto o promedio. Suele ser más estable que un árbol individual, a costa de perder explicación directa.
  • K vecinos más cercanos, KNN: decide a partir de los casos más parecidos. Como usa distancias, las variables deben estar en escalas comparables.
  • Naive Bayes y máquinas de vectores de soporte, SVM: son alternativas frecuentes de clasificación, especialmente en texto o cuando hay muchas variables.
  • K Means y DBSCAN: agrupan observaciones sin target. K Means busca una cantidad fijada de grupos; DBSCAN detecta zonas densas y puede dejar casos aislados como ruido.
  • Redes neuronales: modelan relaciones complejas y son la base de Deep Learning. Suelen necesitar más datos y cómputo, y son menos transparentes.

Ejemplos de candidatos: para spam pueden probarse regresión logística, Naive Bayes o SVM; para precio, regresión lineal, árboles o Random Forest; para segmentar clientes, K Means o DBSCAN. La elección final se hace con validación y con la métrica definida para el problema.

Separación de datos

Evaluar con los mismos ejemplos usados para aprender daría un resultado engañoso. Por eso el dataset se divide antes de entrenar:

  • Entrenamiento o train: datos con los que el algoritmo ajusta los parámetros del modelo.
  • Validación: datos usados para comparar modelos y configuraciones durante el desarrollo.
  • Prueba o test: datos reservados para una evaluación final e independiente. También se lo llama holdout.

Si el test se consulta repetidamente y se toman decisiones a partir de él, deja de medir el comportamiento sobre datos realmente no vistos.

80/20 no es una ley. Es una proporción habitual. La división depende del volumen, la temporalidad, el desbalance, los grupos y el costo de reservar datos. En series temporales no debe romperse el orden causal.

Hiperparámetros

Los parámetros son valores que el modelo aprende de los datos. Los hiperparámetros son configuraciones elegidas antes de entrenar, como la profundidad de un árbol, la cantidad de árboles o el tamaño de cada ajuste en una red. Cambiarlos puede modificar desempeño, tiempo de entrenamiento, costo y tendencia a memorizar ruido.

Grid search prueba una lista fija de combinaciones, random search toma combinaciones al azar y la optimización bayesiana usa los resultados anteriores para decidir qué probar después.

El ajuste suele producir mejoras modestas, por ejemplo pasar de 78 a 78,5 o 79, no necesariamente saltos grandes. Su valor también puede estar en lograr mayor estabilidad, menor costo o menos sobreajuste, no sólo en elevar una métrica.

Validación cruzada o K-fold cross validation

Una única partición de validación puede favorecer o perjudicar al modelo por azar. La validación cruzada divide el conjunto de entrenamiento en K partes, llamadas folds. Se entrena K veces: en cada repetición se usa un fold para validar y los restantes para entrenar. Así se obtiene una distribución de resultados en lugar de una sola cifra.

Son habituales entre 3 y 10 folds. Un K mayor deja menos datos en cada validación y aumenta el costo de cómputo. La elección depende del volumen y de la heterogeneidad de los datos.

Ejemplo: en un dataset de 24.000 ventas dividido en 10 folds, nueve resultados entre 0,70 y 0,74 y uno de 0,41 indican que ese fold contiene un patrón distinto. Podría concentrar Black Friday, una región o un período excepcional. La acción correcta es investigar qué casos lo componen antes de promediar y seguir.

  • Media, interpretación operativa: indica cuán bien o cuán mal el modelo discrimina los patrones que encontró.
  • Desviación estándar, interpretación operativa: indica si encontró todos los patrones. Una dispersión alta señala que algún fold contiene un comportamiento distinto y obliga a investigar su distribución.
Límite de interpretación. Una desviación baja demuestra estabilidad entre los folds observados, no que estén presentes todos los patrones posibles del mundo ni que el modelo los haya aprendido. La lectura de la desviación como señal de cobertura es una regla operativa para detectar heterogeneidad, no una demostración matemática de cobertura total.

Heterogeneidad de patrones

Los patrones pueden cambiar con el tiempo o concentrarse en segmentos. Pandemia, Black Friday o un cambio de sistema son cortes que pueden justificar modelos separados. En series temporales se valida respetando el orden temporal. Si existen clientes, pacientes o equipos relacionados, esos grupos no deben quedar repartidos entre entrenamiento y validación.

Una clasificación puede cometer dos tipos de error: marcar como positivo algo que no lo era, falso positivo, u omitir un positivo real, falso negativo. La matriz de confusión cuenta los cuatro resultados posibles.

Métricas de regresión

  • MAE: promedio del valor absoluto de los errores. Se interpreta en la misma unidad del target.
  • RMSE: raíz del promedio de errores al cuadrado. Penaliza con más fuerza los errores grandes.
  • R cuadrado: proporción de variabilidad explicada respecto de predecir siempre el promedio. Puede ser negativa en datos nuevos.
  • MAPE: error porcentual absoluto medio. Es intuitivo, pero falla o se distorsiona cuando los valores reales son cero o muy pequeños.

Cómo elegir una métrica

  1. Definir cuál es la clase positiva y quién usa la decisión.
  2. Asignar consecuencias a FP y FN.
  3. Revisar qué proporción pertenece a cada clase y si una de ellas domina ampliamente.
  4. Elegir una métrica principal, métricas complementarias y el valor de corte usado para decidir.
  5. Traducir el resultado a costo o valor de negocio.

Ejemplo numérico: sobre el mismo dataset, un algoritmo obtuvo recall de 85% y accuracy de 40%, mientras un modelo de Gradient Boosting, que combina árboles de decisión de forma secuencial, obtuvo accuracy de 81% y recall de 35%. Si el problema exige detectar la mayor cantidad de positivos reales, gana el primero. Si se hubiera elegido accuracy sin pensar el costo de error, se habría elegido Gradient Boosting. El algoritmo ganador cambia con la métrica.

Respuesta: se separan entrenamiento, validación y test. La validación cruzada permite comparar modelos y configuraciones con varias particiones del entrenamiento. La métrica depende del costo de error: recall si importa evitar falsos negativos, precision si importan los falsos positivos y F1 si se necesita equilibrar ambos.

Generalización, sobreajuste y regularización

Generalizar significa mantener un desempeño razonable sobre datos nuevos, no vistos durante el entrenamiento. Si el modelo no aprende suficiente, hay underfitting. Si se adapta demasiado al entrenamiento y falla afuera, hay overfitting. La regularización reúne técnicas que controlan esa adaptación. Bias es el error sistemático de un modelo demasiado rígido, mientras variance es su sensibilidad a los cambios de muestra.

  • Underfitting, modelo demasiado simple: error alto en entrenamiento y validación. No captura la señal suficiente y suele asociarse con sesgo alto.
  • Buen balance: errores bajos y cercanos, estabilidad razonable y comportamiento consistente en producción.
  • Overfitting, modelo que memoriza: error muy bajo en entrenamiento y peor en validación o producción. Suele asociarse con varianza alta.

Dos fuentes frecuentes de sobreajuste

  1. Dataset sesgado o poco representativo: el modelo aprende bien los pocos patrones disponibles, pero el mundo real contiene otros.
  2. Capacidad o entrenamiento mal calibrados: el modelo se adapta al detalle y ruido, o intenta representar demasiados patrones con una arquitectura y presupuesto inadecuados.

Una red neuronal funciona como una base de datos sofisticada: no guarda filas, sino la abstracción de los patrones en sus pesos. Entrenar significa encontrar los pesos o umbrales de activación que permiten propagar la señal correcta y estabilizar la red. El ejemplo de las sillas muestra la idea de abstracción: una persona no memoriza todas las sillas vistas, conserva rasgos generales que le permiten reconocer una nueva.

La capacidad de patrones es finita para una arquitectura y un presupuesto dados. Agregar neuronas o capas amplía capacidad, pero el tiempo necesario para estabilizar la red puede crecer de forma muy fuerte y, según la complejidad del modelo, exponencial. El encadenamiento de redes divide el problema: una primera red reconoce metaconceptos, por ejemplo “P o Q”, y otra red especializada resuelve esa distinción con menos patrones simultáneos.

Definición empírica: Deep Learning combina buenas prácticas de preprocesamiento con un encadenamiento particular de una o más redes neuronales para obtener resultados precisos. Técnicamente, la definición más general es el aprendizaje con redes neuronales de múltiples capas que construyen representaciones jerárquicas. La primera formulación describe una práctica de diseño, la segunda una definición estándar.

Una métrica extremadamente alta no prueba sobreajuste, pero debe activar revisión. Comparar entrenamiento, validación, cross validation y producción permite distinguir un problema fácil de una falsa seguridad. En Iris, un dataset pequeño, estándar y muy curado, una accuracy cercana a 97% es esperable. Obtener el mismo resultado en un dataset real, ruidoso y recién construido exige una revisión mucho más cuidadosa.

Compromiso bias y variance

Un modelo muy simple hace supuestos fuertes y tiene bias alto. Uno muy flexible responde demasiado a cambios de muestra y tiene variance alta. Regularizar significa introducir restricciones o mecanismos que favorecen soluciones que generalizan, aceptando a veces una pequeña pérdida en entrenamiento.

Si un modelo obtiene 96% en entrenamiento, 95% en test y cae a 75% al desplegar, el test puede no representar el entorno, puede haber leakage, puede haber cambiado la distribución o puede existir sobreajuste. Es una señal para investigar, no un diagnóstico único.

Explicabilidad de modelos y detección de sesgos (XAI)

De caja negra a explicación

XAI, Explainable Artificial Intelligence, reúne técnicas para entender qué variables influyeron en un modelo y por qué produjo una salida. Un modelo es interpretable de forma inherente cuando su funcionamiento puede inspeccionarse directamente, como un árbol pequeño. En una caja negra, la explicación se construye después de predecir. SHAP, por ejemplo, reparte la diferencia entre una predicción y un valor de referencia entre las variables de entrada.

Explicar no convierte automáticamente al modelo en correcto, causal o justo. Solo aporta evidencia sobre su comportamiento.

  • Explicación global: muestra qué variables y relaciones dominan el comportamiento promedio del modelo.
  • Explicación local: muestra por qué una observación particular recibió esa salida.
  • Explicación contrafáctica: muestra qué cambio mínimo en la entrada habría modificado la decisión.

Herramientas mencionadas

Lectura de SHAP: en una predicción de 0,70 sobre el mejor jugador de un partido, una variable como goles puede empujar el resultado hacia arriba y otra como pocos pases hacia abajo. Por qué XAI importa

  • Permite revisar si el modelo usa señales plausibles o atajos.
  • Facilita la validación con el owner y los expertos de dominio.
  • Ayuda a descubrir reglas de negocio y formalizar conocimiento tácito.
  • Soporta auditoría, reclamos, mejora y cumplimiento.
  • Guía nuevas iteraciones de features y pruebas de sesgo.

ML, dinámica de sistemas y conocimiento

La dinámica de sistemas de Jay Forrester y el pensamiento sistémico difundido por Peter Senge modelan interacciones y retroalimentaciones entre procesos. Un modelo de ML puede predecir sin representar esas relaciones. Las explicaciones validadas por expertos deben documentarse como conocimiento o reglas de negocio, pero no prueban causalidad.

Sesgo en datos y modelo

Los datos históricos reflejan decisiones y desigualdades anteriores. Un algoritmo puede reproducirlas o amplificarlas. También puede usar un proxy, una variable que revela indirectamente otra, como el código postal respecto del nivel socioeconómico, aunque se elimine una variable sensible. Por eso no alcanza con mirar importancia sobre entrenamiento.

Proceso de prueba

  1. Identificar grupos, variables sensibles y daños posibles.
  2. Revisar representación, etiquetas, calidad y proxies.
  3. Construir conjuntos de prueba y pares contrafácticos.
  4. Comparar tasas de error y resultados entre grupos.
  5. Analizar explicaciones locales y globales.
  6. Mitigar en datos, features, objetivo, umbral o algoritmo.
  7. Documentar límites, pruebas y decisión residual.
  8. Monitorear en producción, porque la equidad también deriva.
Caso de crédito. Una mujer reunía el mismo perfil que solicitantes aprobados, pero el modelo la rechazaba porque había aprendido sexo y raza como atajos históricos. El algoritmo no posee intención, pero el equipo sigue siendo responsable por dataset, objetivo, variables admitidas, pruebas, despliegue y consecuencias.

Explicabilidad no es equidad

Una explicación muestra cómo decidió el modelo. La equidad, también llamada fairness, evalúa si el comportamiento satisface un criterio aceptable entre grupos. Se relacionan, pero no son equivalentes. Un modelo puede ser explicable y discriminatorio, o menos interpretable y mostrar mejores métricas de equidad.

Respuesta: XAI atribuye influencia a las variables de forma global o local. Las pruebas de sesgo comparan predicciones, errores y resultados entre grupos o contrafácticos. Una explicación puede ayudar a detectar una variable indirecta o proxy, pero no demuestra equidad.

Producción y mantenimiento de modelos (MLOps)

MLOps, abreviatura de Machine Learning Operations, es el conjunto de prácticas para versionar, desplegar, observar y mantener modelos de aprendizaje automático en producción. Conecta el trabajo de datos y modelado con ingeniería de software y operación. No es una herramienta específica ni empieza después de entrenar.

Por qué un buen modelo no llega a producción

Cifra de referencia: alrededor del 40% de los proyectos que llegan validados nunca alcanzan producción por errores u horrores de gestión. No debe tomarse como una estadística universal, sino como advertencia para no separar el modelado de la planificación productiva.

  • Latencia: la aplicación que lo integra ya tiene un contrato de respuesta. El modelo debe cumplirlo con carga real.
  • Seguridad: datos, dependencias, acceso, interfaz, respuestas y archivos producidos por el entrenamiento requieren controles.
  • Cómputo: el entrenamiento viable no garantiza inferencia viable por costo, memoria, aceleradores o concurrencia.

Latencia, seguridad, costo, integración y capacidad de cómputo deben evaluarse durante la viabilidad, no después del entrenamiento.

Qué debe registrarse

  • Versión de código, modelo, dataset, features y configuración.
  • Fecha, parámetros, hiperparámetros, métricas y archivos producidos por el experimento.
  • Entradas y salidas productivas con controles de privacidad.
  • Latencia, errores, uso de memoria, procesador general, CPU, procesador gráfico, GPU, y disponibilidad.
  • Distribuciones de entrada y salida, y desempeño cuando llega el valor real confirmado, llamado ground truth.
  • Decisiones de aprobación, reentrenamiento, reversión y retiro.

MLflow y versionado

MLflow es una herramienta para registrar experimentos, parámetros, métricas y archivos resultantes, comparar ejecuciones y reconstruir modelos. El mejor modelo no necesariamente es el último: después de muchas iteraciones, la ejecución óptima puede haber quedado en el medio y debe poder recuperarse. Git versiona bien código, pero los datasets suelen necesitar estrategias específicas por su tamaño. La trazabilidad completa vincula código, datos, entorno y ejecución.

Cambios en producción o drift

Drift significa que el entorno productivo dejó de parecerse al representado durante el desarrollo. El cambio puede estar en las entradas, en la relación con el target, en las predicciones o en el propio pipeline.

TipoQué cambiaEjemplo

Cierre responsable

El retiro incluye desactivar servicios de predicción y tareas automáticas, archivar versiones y evidencia, resolver dependencias, comunicar a usuarios, revisar obligaciones de retención y registrar lecciones. Solo entonces el proyecto queda cerrado.

Respuesta: MLOps versiona código, datos, modelos y configuración, automatiza despliegues y registra operación. El monitoreo mide drift de entradas, relación con el target, predicciones y desempeño para decidir si corresponde corregir el pipeline, reentrenar, revertir o retirar el modelo.

Automatización del aprendizaje automático (AutoML)

AutoML, Automated Machine Learning, reúne herramientas que automatizan partes repetitivas del entrenamiento y la comparación de modelos. Suele recibir un dataset preparado, el target, el tipo de problema, la métrica, la estrategia de validación y un presupuesto de tiempo. Como salida produce resultados comparables y uno o más pipelines candidatos. Un pipeline es una secuencia ordenada de transformaciones y modelo.

No significa que el proyecto de ML sea automático. Su primer valor suele ser generar un baseline, una referencia inicial contra la cual medir mejoras posteriores.

AutoML reduce el trabajo operativo necesario para entrenar y comparar muchos algoritmos. El conocimiento no desaparece: se desplaza hacia la definición del problema y la interpretación de las métricas.

Tres bloques que puede automatizar

  • Preparación y features: aplica preprocesamientos estandarizados y, según la herramienta, selecciona o genera características.
  • Modelo e hiperparámetros: compara algoritmos, prueba configuraciones y los evalúa con validación cruzada usando la métrica indicada.
  • Combinación de modelos: puede crear ensambles, es decir, usar varios modelos y combinar sus predicciones. Bagging los entrena en paralelo y vota o promedia, boosting los entrena en secuencia para corregir errores, y stacking aprende a mezclar modelos distintos.

Además puede registrar métricas, generar gráficos y exportar modelos, pipelines o código. La recolección, integración y limpieza de los datos crudos ocurren antes: AutoML recibe un dataset cuya calidad ya debe haber sido investigada.

Qué no delega

Definición del problema, acceso y calidad, prevención de leakage, métrica adecuada, separación de los datos, validación con expertos, análisis de sesgos, restricciones de producción, monitoreo y responsabilidad.

Criterios para seleccionar herramienta

Actividad del proyecto, licencia, reproducibilidad, integración, seguridad, capacidad de exportar, soporte de tipos de problema, observabilidad, costo y dependencia en producción. Una herramienta open source abandonada puede ser adecuada para aprender, pero riesgosa como base corporativa.

TPOT y seguridad: el CISO, responsable de seguridad de la información, puede autorizar librerías distintas para desarrollo, pruebas internas, validación de usuarios y producción. TPOT puede utilizarse en un entorno de desarrollo más flexible y exportar un pipeline basado en librerías estándar, sin necesitar TPOT en producción.

Respuesta: AutoML automatiza comparación de algoritmos, pipelines, validación y ajuste. No define el problema, no corrige datos deficientes y no decide métricas, sesgos aceptables ni restricciones productivas.

Modelos de lenguaje, Transformer y prompting

Un LLM, Large Language Model, es una red neuronal de Deep Learning entrenada con grandes cantidades de texto para predecir tokens y generar una continuación. Recibe un prompt, que contiene instrucciones y contexto, y produce una respuesta. No funciona como una base de datos ni garantiza que una afirmación sea verdadera.

Cómo procesa texto un LLM

  1. Tokenización: divide el texto en tokens, que pueden ser palabras, partes de palabras o signos.
  2. Embeddings: convierte cada token en un vector numérico que el modelo puede procesar.
  3. Posición: agrega información sobre el orden, porque cambiar el orden puede cambiar el significado.
  4. Atención: asigna más peso a las partes del contexto relevantes para cada token y procesa muchas relaciones en paralelo. Así distingue sentidos diferentes de una palabra, como cabo geográfico, militar o náutico, según las palabras que la rodean.
  5. Predicción: después de atravesar varias capas Transformer, calcula probabilidades para el token siguiente. Repite el proceso token por token hasta completar la respuesta.

El Transformer es la arquitectura de red neuronal que organiza ese mecanismo de atención. Durante el preentrenamiento ajusta sus pesos con grandes corpus de texto. En inferencia usa esos pesos para responder. Un fine tuning posterior puede especializar su conducta, pero no convierte al modelo en una base de datos verificable.

Cómo escribir una instrucción útil

Un prompt es el texto con el que se le indica una tarea al modelo. La ingeniería de prompts consiste en redactar y refinar esas instrucciones para obtener una salida útil. Un prompt necesita claridad, detalle suficiente y contexto. El esquema ROC ayuda a ordenarlo: rol que debe asumir el modelo, objetivo concreto y contexto necesario para resolverlo.

Para generar un pipeline de ML, por ejemplo, conviene pedir por separado las features y el target, variables de entrada, transformaciones o ecuaciones, algoritmo e hiperparámetros, y pasos de implementación y prueba. El primer resultado se revisa y el prompt se ajusta. Sin conocimiento del problema no hay forma de especificar ni controlar la salida.

Respuesta: un LLM tokeniza la entrada, la representa con vectores y usa atención dentro de una arquitectura Transformer para ponderar el contexto y predecir el token siguiente. Un prompt útil explicita rol, objetivo, contexto, formato y criterios de aceptación, y luego se refina según el resultado.

Vibe coding y supervisión humana

Vibe coding es el desarrollo de software asistido por modelos de lenguaje mediante instrucciones en lenguaje natural. Los tokens procesados determinan tanto el límite de contexto como una parte importante del costo.

La analogía del ilusionista describe el vibe coding: el resultado hace pensar que hay comprensión detrás de la pantalla, pero el mecanismo sigue siendo probabilístico. Que el código compile no demuestra que la jerarquía, los contratos, la seguridad o la mantenibilidad sean correctos.

Flujo de trabajo

  1. Cargar el documento base o corpus, conjunto de documentos de trabajo, en NotebookLM y extraer conceptos, preguntas y referencias.
  2. Actualizar y contrastar la búsqueda con Gemini y Consensus, preferentemente sobre papers en inglés.
  3. Extraer dataset, variables, ecuaciones, arquitectura y pipeline.
  4. Crear un documento técnico en Markdown como especificación.
  5. Dar al editor asistido reglas, contexto y criterios de prueba.
  6. Generar una primera versión ejecutable con datos sintéticos.
  7. Registrar decisiones, errores y ejecuciones en una bitácora como memory.md.
  8. Leer, probar y refinar el código con conocimiento de ingeniería.

Ejemplo de costos

Riesgos

  • Amnesia de código: el equipo deja de entender estructura y decisiones, y solo puede modificar el sistema volviendo a pedirle todo al modelo.
  • Promedio funcional: el código generado suele reflejar patrones frecuentes. Puede compilar sin ser eficiente, seguro, mantenible o bien diseñado.
  • Colapso de diversidad: si futuros modelos se entrenan con código producido por modelos anteriores, se acentúa la media y disminuye la desviación. El resultado tiende a soluciones cada vez más parecidas, con menos alternativas de diseño y menor evolución.
  • Costo de contexto: pasar repositorios enteros consume tokens, expone información y vuelve impreciso el alcance. El costo depende principalmente de la cantidad de tokens necesaria para comunicar todo el código y sus relaciones, no del precio unitario de cada token.
  • Falsa confianza: la analogía con Dunning Kruger advierte que una respuesta segura no demuestra comprensión ni corrección.

El costo de combinar programadores senior con grandes consumos de tokens puede igualar o superar el esquema previo. La productividad aparece cuando se reduce el contexto necesario y se usa a las personas para diseñar y revisar, no cuando se entrega el repositorio completo al modelo.

Patrones como compresión de contexto

Un patrón es una solución probada a un problema recurrente. El trabajo sigue un embudo, de lo general a lo particular: primero arquitectura, luego patrones de diseño, módulos y contratos, y recién después código. Ese scaffolding o andamiaje permite pedir cambios acotados, reducir contexto y evaluar el resultado. Para código legado, primero puede ser necesario refactorizar y documentar límites.

Para que el vibe coding funcione de manera consistente hay que pasarle al modelo los patrones con los que tiene que trabajar. El resultado es software menos promedio, más modular y revisable.

Dunning Kruger y sesgo heredado

El efecto Dunning Kruger describe la tendencia a sobreestimar el propio dominio cuando el conocimiento es superficial. El ladrón que se puso jugo de limón en la cara porque creyó que eso lo volvería invisible para las cámaras ilustra el sesgo: el error no era falta de seguridad, sino una explicación falsa sostenida con certeza.

Los LLM suelen responder con tono seguro aunque la respuesta sea incompleta o incorrecta. Ese riesgo de que la persona adopte la certeza del modelo como propia puede llamarse sesgo heredado. El antídoto es conocimiento de base suficiente para interpelar al modelo, repreguntar, comparar alternativas y revisar el resultado.

RAG (generación aumentada mediante recuperación) y bases vectoriales

RAG, Retrieval Augmented Generation, combina recuperación y generación. Busca fragmentos en un corpus, colección de documentos, y los agrega al prompt del LLM para que responda con esa evidencia.

Cada fragmento o chunk puede guardarse junto con metadata, como fuente, fecha y permisos. Un embedding lo representa como un vector numérico para buscar contenido semánticamente cercano.

Las tres limitaciones del LLM y las tres ventajas de RAG

  1. Fine tuning costoso: el fine tuning es un entrenamiento adicional que modifica el modelo para especializarlo. Requiere datos, cómputo y un nuevo proceso de ajuste. RAG: incorpora conocimiento mediante documentos e índice sin volver a entrenar todo el modelo.
  2. Obsolescencia: el conocimiento guardado en los pesos queda atrasado. RAG: se actualiza cambiando documentos y reindexando, sin volver a entrenar todo el modelo.
  3. Opacidad: es difícil reconstruir por qué los pesos produjeron una respuesta. RAG: permite trazar qué fragmentos y documentos fueron recuperados.

Pipeline básico

  1. Ingestar documentos confiables.
  2. Limpiar, dividir en fragmentos o chunks y agregar metadata.
  3. Convertir los chunks en embeddings.
  4. Indexar los vectores.
  5. Recuperar los vecinos relevantes para la consulta.
  6. Generar la respuesta con contexto y fuentes.

Embeddings y similitud

Un embedding representa contenido en un vector donde cercanía pretende capturar similitud semántica. La consulta se embebe con el mismo modelo y se buscan vecinos mediante similitud coseno, producto interno o distancia euclídea. PCA puede visualizar en pocas dimensiones, pero no forma parte necesaria de la búsqueda.

La similitud coseno compara la orientación de dos vectores, la distancia euclídea mide la distancia recta entre ellos y el producto interno combina orientación y magnitud. La medida correcta depende de cómo fue entrenado y normalizado el modelo de embeddings.

Por qué la base vectorial siempre responde

  1. Una consulta exacta en una base de datos relacional, escrita en SQL, puede no devolver filas. Una búsqueda vectorial es difusa: ordena los fragmentos por cercanía semántica.
  2. El índice siempre puede devolver el vector más cercano, aunque ese vecino siga siendo poco relevante.
  3. No existe un punto de corte universal y verificado que separe una respuesta correcta de una incorrecta. Un umbral elegido sin evaluación para ese sistema es arbitrario.
  4. La base vectorial siempre responde y, si lo recuperado no es lo que se necesitaba, se lo llama alucinación.

En una implementación real: pueden configurarse umbrales, abstención, filtros y un segundo ordenamiento. Eso reduce recuperaciones malas, pero ningún valor garantiza verdad para todas las consultas. Hay que distinguir recuperación irrelevante de alucinación generativa: la base recupera y el LLM redacta.

Por qué usar RAG

  • Actualiza conocimiento cambiando documentos e índice, sin reentrenar todo el LLM.
  • Restringe el contexto a fuentes empresariales.
  • Permite citar y auditar documentos recuperados.
  • Reduce costo frente a ciertos ajustes o entrenamientos.

Limitaciones

RAG no garantiza corrección. Puede fallar por documentos malos, fragmentación inadecuada, embeddings poco aptos, consulta ambigua, metadata ausente, una cantidad incorrecta de resultados recuperados, llamada top K, pérdida de contexto, prompt deficiente o generación que ignora la evidencia. Deben evaluarse tanto la recuperación como la respuesta final.

RAG frente a fine tuning

Fine tuning es un entrenamiento adicional que modifica pesos o adaptadores de un modelo para especializar su comportamiento. RAG no modifica el modelo: cambia el contexto que recibe en cada consulta.

RAGFine tuning

Respuesta: RAG recupera fragmentos mediante embeddings y los agrega al contexto del LLM. El corpus se actualiza reindexando, sin reentrenar el modelo. Deben evaluarse por separado recuperación, fidelidad a las fuentes, citas, latencia y seguridad.

Principales brechas de seguridad en modelos de lenguaje (LLM)

Las cinco que hay que nombrar primero

  1. Prompt injection: una entrada directa o un contenido externo introduce instrucciones que cambian el comportamiento previsto. Puede revelar datos, alterar decisiones o disparar herramientas.
  2. Divulgación de información sensible: el sistema expone PII (Personally Identifiable Information, información personal identificable), secretos, datos empresariales, fragmentos de entrenamiento, contexto de otros usuarios o credenciales.
  3. Envenenamiento y cadena de suministro: datos, embeddings, modelos, adaptadores, dependencias o repositorios comprometidos insertan sesgos, puertas traseras o código malicioso.
  4. Manejo inseguro de salidas: la aplicación confía en texto generado y lo ejecuta o muestra sin validar, habilitando comandos, consultas SQL o scripts maliciosos en páginas web, llamados XSS.
  5. Agencia excesiva: agencia es la capacidad del sistema de ejecutar acciones mediante herramientas. Si recibe más permisos o autonomía que la necesaria, una alucinación o inyección puede producir una acción real dañina.
El LLM no es un control de seguridad. Un prompt no reemplaza autorización, validación, aislamiento ni permisos mínimos. Los controles críticos deben ser determinísticos y externos al modelo.

Caso OpenClaw: durante 15 a 20 días fue adoptado con entusiasmo como asistente capaz de planificar y ejecutar tareas, hasta que se hicieron visibles sus agujeros de seguridad. Estos sistemas necesitan permisos acotados y aprobación humana para las acciones sensibles.

  • Directa: el usuario escribe una instrucción hostil en la conversación, por ejemplo para ignorar políticas o revelar contexto.

  • Poisoning del corpus: documentos maliciosos manipulan futuras respuestas.

Agentes inteligentes y orquestación

Agente: programa que recibe información del entorno y ejecuta acciones para cumplir un objetivo dentro de permisos definidos.

Orquestador: componente que decide qué agente o herramienta usar, en qué orden, con qué contexto y ante qué condiciones. Los agentes por sí solos no saben nada: su valor depende de capacidades bien definidas y de la coordinación.

n8n es un ejemplo de orquestador visual: permite dibujar una secuencia de pasos, condiciones y conexiones entre herramientas.

Integración empresarial: Machine Learning o Deep Learning detecta y predice patrones, RAG aporta conocimiento documental actualizable y trazable, y los agentes ejecutan tareas coordinadas por un orquestador. La combinación integra modelado, conocimiento documental y ejecución controlada.

Propiedades

  • Autonomía: opera con intervención limitada y decide dentro de permisos definidos.
  • Reactividad: responde oportunamente a cambios del entorno.
  • Proactividad: toma iniciativa y planifica hacia objetivos.
  • Habilidad social: se comunica con personas, herramientas u otros agentes.

Tipos clásicos

Sistemas multiagente

Dividen un problema en tareas acotadas y agentes especializados. Los contratos de entrada y salida permiten reemplazar una pieza sin rehacer todo. La modularidad ayuda, pero introduce coordinación, observabilidad, seguridad, estado compartido y fallas distribuidas.

Orquestación y colaboración

  • Orquestación, flujo predefinido: una persona o flujo de trabajo define primero A, luego B, condiciones y errores. n8n y Zapier son ejemplos de automatización por eventos.
  • Colaboración, plan dinámico: un agente de nivel superior interpreta el objetivo, selecciona capacidades y arma o revisa el plan durante la ejecución.

La memoria compartida puede implementar un patrón Blackboard: agentes leen y escriben estado común. El orquestador coordina, pero debe evitar que datos viejos, permisos excesivos o una salida errónea contaminen todo el flujo.

MCP

MCP significa Model Context Protocol. Estandariza cómo aplicaciones de IA descubren y usan herramientas o recursos. No vuelve inteligentes a los agentes ni resuelve por sí solo planificación, identidad, autorización o seguridad.

Asistente sobre orquestador

Un asistente planificador recibe el problema, identifica agentes disponibles y construye el plan. El agente ejecuta una capacidad específica y el orquestador coordina el flujo.

Riesgo productivo. Más autonomía exige más límites: permisos mínimos, ejecución aislada, aprobación humana para acciones sensibles, presupuestos, tiempos máximos, validación de resultados, registro de actividad y capacidad de detener o revertir.

Respuesta: un agente recibe información y ejecuta acciones con cierta autonomía. Un sistema multiagente reparte tareas entre agentes especializados. Un orquestador ejecuta un flujo definido, mientras un planificador puede seleccionar capacidades y modificar el plan durante la ejecución.

Aplicaciones industriales de Machine Learning

Cobots, mantenimiento predictivo y gemelos digitales son casos de aplicación. Combinan percepción, predicción, simulación y decisión sobre procesos físicos; no constituyen nuevas familias de aprendizaje automático.

Robot industrial frente a cobot

  • Robot tradicional: opera en tarea y espacio controlados, frecuentemente fijo y separado por jaulas. La seguridad depende de aislamiento y secuencias previsibles.
  • Cobot: está diseñado para compartir espacio con personas mediante percepción, limitación de fuerza, sensores y control adaptativo. Colaborativo no significa seguro por defecto.

Un brazo tradicional que no percibe contacto puede continuar su movimiento y aplastar una mano. La colaboración exige detectar personas, contacto y fuerza, además de limitar el movimiento. En tareas como colocar un tablero de automóvil, el cobot puede sostener el peso mientras la persona realiza el ajuste fino.

Periodización de generaciones

  1. 1950 a fines de los 60: controles hidráulicos, sensores básicos y jaulas de seguridad.
  2. Década de 1970 hasta aproximadamente 1999: auge de los controladores lógicos programables, PLC, y de los robots programables, todavía fijos y con poco o ningún sentido del entorno. La línea debía llevar cada pieza a una posición exacta.
  3. Desde el año 2000, cuarta generación: comienza el concepto de cobot, con autopercepción, sensores de fuerza y detección de personas para colaborar en un espacio compartido.
  4. Últimos cinco años de la periodización, quinta generación: nube, mayor cómputo, movilidad, Deep Learning e IA generativa para una adaptación más rica.

Estas fechas sirven como periodización de la materia, pero la denominación por generaciones no constituye una taxonomía universal de robótica.

Capacidades destacadas

  • Sensores de fuerza, contacto y percepción del entorno.
  • Visión artificial no invasiva: permite percibir personas, piezas, posiciones, fugas, temperatura y defectos sin llenar el entorno de sensores de contacto. Un robot necesita este alto grado de percepción para ser considerado cobot.
  • Movilidad y gestión de energía en equipos desconectados.
  • Kinetic teaching: una persona guía movimientos y el sistema ajusta la rutina.
  • Asistencia ergonómica, por ejemplo sostener el peso de una pieza mientras el operario la posiciona.
  • Conectividad y cómputo local o remoto para modelos más complejos.

Mantenimiento

EstrategiaPreguntaAcción
Reactivo¿Qué se rompió?Intervenir después de la falla.
Preventivo¿Cuándo toca por calendario o uso?Intervenir periódicamente.
Predictivo¿Qué probablemente fallará y cuándo?Anticipar con condición y datos.

Predictivo es la bola de cristal: anticipa que el equipo fallará y cuándo. Prescriptivo es la receta: agrega qué acción realizar para evitar la falla o transitarla de forma controlada.

  • MTBF, Mean Time Between Failures: tiempo medio entre fallas de un equipo. Ayuda a caracterizar su confiabilidad y planificar intervenciones.
  • OEE, Overall Equipment Effectiveness: efectividad general del equipo, normalmente vinculada con disponibilidad, rendimiento y calidad. La cifra de referencia es 89% de efectividad, interpretada como operación sin paradas imprevistas, con detenciones predichas y planificadas.

Gemelo digital

Es una representación digital conectada con un activo o proceso real que modela comportamiento, estado e interacción, no solo apariencia. Permite simular desgaste, consumo, fallas y configuraciones antes de intervenir el sistema físico. Con datos sintéticos puede acelerar entrenamiento y pruebas, pero necesita calibración contra realidad.

Esta simulación puede vincularse con las PINN, Physics Informed Neural Networks, redes cuyo entrenamiento incorpora restricciones físicas conocidas, como gravedad, termodinámica o conservación. Esas leyes limitan comportamientos imposibles del gemelo digital y ayudan a que los datos sintéticos respeten el sistema real.

Infraestructura y límites

La inteligencia del modelo no reemplaza la infraestructura. Una fábrica con integración vertical, como el ejemplo de Tesla, puede coordinar datos y automatización de forma distinta de una cadena fragmentada. Llevar cómputo al espacio ofrece energía solar y cobertura, pero introduce radiación, disipación de calor y comunicaciones como restricciones. Starlink aplicado a drones durante la guerra entre Ucrania y Rusia muestra que conectividad, cifrado y velocidad pueden ser tan determinantes como el modelo.

La figura del gato japonés maneki-neko se usó para mostrar la abstracción humana: una persona puede relacionar un objeto nuevo con cultura, intención y experiencias de otros dominios. Un modelo extrapola desde patrones aprendidos y no realiza esa transferencia abierta con la misma flexibilidad.

Respuesta: un cobot comparte espacio con personas y limita riesgo mediante sensores, control de fuerza y percepción. ML puede estimar fallas o adaptar movimientos. Un gemelo digital simula el activo para probar configuraciones y modelos antes de aplicarlos al sistema físico.

Formato del final: examen teórico, escrito y presencial. El trabajo práctico debe estar aprobado antes de rendir y, si se presenta en una fecha de examen, el final se rinde en una fecha posterior. Aproximadamente 40% puede evaluar ejemplos y conceptos que no aparecen en las diapositivas.

Simulacros basados en los enunciados reales

A. Describa las tres familias de algoritmos de ML

Respuesta: las tres familias son clasificación, regresión y clustering. Clasificación es supervisada: aprende con un target categórico y devuelve una clase o probabilidad, por ejemplo falla o no falla. Regresión también es supervisada: el target es continuo y devuelve un valor, por ejemplo ventas o vida útil restante. Clustering es no supervisado: no recibe target y busca grupos o estructuras, por ejemplo segmentos de clientes o regímenes de operación. En una taxonomía por paradigma, clasificación y regresión quedan dentro de supervisado, clustering dentro de no supervisado, y aprendizaje por refuerzo es otro paradigma. Se habla de familias porque no se conoce de antemano la distribución interna de los patrones: dos datasets con el mismo objetivo y las mismas columnas pueden requerir algoritmos distintos de la misma familia. AutoML permite compararlos.

B. Dado un dataset de ventas de e-commerce, ¿qué se puede hacer con ML?

Primero no elegiría un algoritmo, elegiría una decisión. El mismo dataset habilita varios problemas:

  • Regresión o series temporales: predecir unidades o facturación por producto y día para stock.
  • Clasificación: predecir abandono de clientes, devolución, fraude o conversión.
  • Clustering: segmentar clientes por frecuencia, recencia, monto y categorías.
  • Recomendación: ordenar productos relevantes por usuario o sesión.
  • Anomalías: detectar compras, descuentos o devoluciones fuera de norma.

Como caso concreto, para pronosticar demanda definiría como target las unidades vendidas por producto y día, el horizonte temporal y una métrica como MAE. Haría EDA de nulos, devoluciones, promociones, faltantes de stock, Black Friday y cambios de catálogo. Crearía features de calendario, precio, descuento, valores pasados y medias móviles. Separaría temporalmente entrenamiento, validación y test para no usar información futura. Compararía una referencia estacional simple con modelos, explicaría resultados y monitorearía cambios en los datos. Los días promocionales podrían requerir un modelo o régimen separado.

C. Se quiere predecir una falla en un pozo petrolero a 2000 metros, ¿qué haría?

Definición: acordaría qué significa falla, en qué componente y con cuánto anticipo debe alertarse. Sin horizonte no hay target útil. También definiría el costo de falso negativo, falso positivo y parada.

Datos: series temporales de presión, temperatura, caudal, vibración, torque, RPM, energía, propiedades del fluido, profundidad, mantenimiento, configuración y eventos. Validaría sincronización, frecuencia, sensores, unidades, cortes de comunicación y etiquetas de fallas.

Modelado: si existen suficientes fallas etiquetadas, plantearía clasificación para falla dentro de una ventana o regresión para vida útil restante. Si las fallas son muy escasas, complementaría con detección de anomalías no supervisada y conocimiento físico. Separaría por tiempo y, si hay varios pozos, también por pozo para medir generalización real.

Operación: priorizaría recall sin dejar que las falsas alarmas vuelvan inútil el sistema. Validaría con especialistas, simularía con gemelo digital si existe, desplegaría con límites seguros y monitorearía drift de sensores y regímenes. El modelo asesora mantenimiento predictivo. La acción prescriptiva debe considerar seguridad y costo.

D. Hay poca o desconocida distribución de datos, ¿qué familia usaría para conocerla?

Si el enunciado significa que no conozco cómo se organizan los datos y no tengo etiquetas, usaría aprendizaje no supervisado, en particular clustering, reducción de dimensionalidad y detección de anomalías, junto con EDA. Eso ayuda a descubrir grupos y regímenes, no a demostrar una distribución verdadera.

Si “poca distribución” significa pocos datos o baja representatividad, ninguna familia inventa la información faltante. Primero conseguiría más evidencia, usaría modelos simples y regularizados, validación cuidadosa, conocimiento de dominio y, si corresponde, datos sintéticos validados. Deep Learning suele ser una peor primera opción con pocos datos.

E. Enumere paso a paso cómo construir un RAG
  1. Definir dominio, usuarios, fuentes autorizadas, frescura y criterio de éxito.
  2. Ingestar, limpiar y versionar documentos con metadata y permisos.
  3. Dividir en chunks que preserven contexto y relaciones.
  4. Generar embeddings con un modelo consistente.
  5. Guardar vectores, texto y metadata en un índice.
  6. Al consultar, normalizar la pregunta y generar su embedding.
  7. Recuperar top K con filtros de acceso y, si hace falta, búsqueda híbrida.
  8. Reordenar, filtrar irrelevantes y construir el contexto.
  9. Pedir al LLM que responda solo con evidencia, cite y se abstenga si no alcanza.
  10. Evaluar recuperación, fidelidad, respuesta, tiempo de respuesta, seguridad y costo, luego monitorear.
F. Nombre las principales brechas de seguridad en LLM

Nombraría primero prompt injection directa e indirecta, divulgación de datos sensibles, envenenamiento de datos o embeddings, cadena de suministro comprometida, manejo inseguro de salidas y agencia excesiva. Completaría con fuga del prompt de sistema, debilidades vectoriales y de permisos en RAG, desinformación y consumo sin límites. Un prompt no es un control de seguridad: hacen falta mínimo privilegio, aislamiento, validación, cuotas, trazabilidad y aprobación humana para acciones críticas.

Preguntas adicionales de práctica

1. ¿Por qué un proyecto de ML no empieza entrenando?

Porque primero hay que validar pertinencia, criterio de éxito, evidencia previa, datos, acceso, recursos y riesgo. Sin eso se puede optimizar un algoritmo para un problema mal formulado. El EDA completa el estudio de viabilidad y habilita la decisión Go o No Go.

2. ¿Cuál es la diferencia entre EDA, limpieza e ingeniería de características?

EDA diagnostica sin modificar. Limpieza corrige o trata problemas para alcanzar calidad mínima. Feature engineering cambia la representación para favorecer aprendizaje, como descomponer fechas o crear ratios.

3. ¿Qué es un outlier y siempre debe eliminarse?

Es una observación fuera de norma, no necesariamente un error. Puede ser fraude, evento especial o caso válido. Se investiga su origen y se decide según objetivo y algoritmo: mantener, transformar, segmentar, eliminar o modelar aparte.

4. ¿Por qué correlación no implica causalidad?

Dos variables pueden moverse juntas por azar, por una tercera causa o solo durante el período observado. La correlación describe asociación, no mecanismo causal. Sirve para explorar y detectar redundancia, pero requiere contexto y validación.

5. ¿Por qué accuracy puede ser una mala métrica?

Con clases desbalanceadas puede ser alta prediciendo siempre la mayoría. Además trata FP y FN como si costaran igual. En medicina suele priorizarse recall para no omitir casos, y en otros dominios precision para reducir falsas alarmas.

6. ¿Qué aporta K fold cross validation?

Repite entrenamiento y validación rotando K particiones, por lo que cada registro cumple ambos roles. La media indica cuán bien discrimina los patrones encontrados y la desviación estándar indica si se captaron todos los patrones distribuidos entre los folds. Técnicamente, permite estimar desempeño promedio y sensibilidad a la partición, pero no reemplaza un test independiente ni demuestra cobertura de patrones que no están en la muestra.

7. ¿Qué indica una desviación alta entre folds?

Indica que no se encontraron todos los patrones de manera estable. En términos técnicos, muestra sensibilidad a la partición y puede sugerir pocos datos, patrones heterogéneos, grupos, cambio temporal, outliers o un modelo de alta varianza. Exige investigar qué registros o períodos explican la diferencia.

8. ¿Qué es hiperparametrización?

Es buscar valores de configuración externos al aprendizaje para mejorar el sistema. Puede optimizar métrica, estabilidad, regularización, latencia o costo. Debe evaluarse con validación, no sobre el test final.

9. ¿Qué es sobreajuste?

Es aprender entrenamiento y ruido con demasiado detalle, logrando buen resultado allí y peor en datos nuevos. Se detecta comparando curvas y conjuntos, y se mitiga con regularización, simplificación, más datos representativos, early stopping, dropout o augmentation.

10. ¿Cuál es el compromiso bias y variance?

Modelos demasiado simples tienen bias alto y subajustan. Modelos demasiado flexibles tienen variance alta y sobreajustan. El objetivo es elegir complejidad y regularización que minimicen error en datos no vistos.

11. ¿AutoML reemplaza al científico de datos?

No. Automatiza comparación, pipelines y ajuste, pero no garantiza calidad, ausencia de leakage, métrica correcta, equidad ni viabilidad productiva. La definición del problema, los controles y la aprobación siguen a cargo del equipo.

12. ¿Qué diferencia hay entre SHAP y una prueba de sesgo?

SHAP atribuye contribuciones a features en predicciones. Una prueba de sesgo compara resultados y errores entre grupos o contrafácticos bajo un criterio de equidad. La explicación puede ayudar a diseñar la prueba, pero no demuestra equidad.

13. ¿Por qué un modelo no puede entregarse como caja negra?

Porque debe ser validado, auditado y comprendido por quienes asumen riesgo. La explicación detecta atajos, permite contraste con expertos, documenta reglas y ayuda a responder reclamos y mejorar el sistema.

14. ¿Qué bloquea el pasaje a producción?

Especialmente latencia, seguridad y capacidad de cómputo no planificadas. También integración, operación, calidad del servicio, datos online, observabilidad y responsabilidad. Una cifra de referencia es que 40% de los proyectos validados no llegan a producción por errores u horrores de gestión. Estos riesgos deben considerarse desde viabilidad.

15. ¿Qué es drift?

Es el cambio entre el entorno actual y el representado durante entrenamiento. Puede cambiar la distribución de las entradas, la relación entre entradas y resultado, la distribución de predicciones o el pipeline. Se monitorea y diagnostica antes de decidir reentrenamiento.

16. ¿Cuándo termina un proyecto de ML?

No al desplegar. Termina cuando el modelo se retira de forma controlada, se cierran dependencias, se archivan evidencias, se documentan resultados y lecciones y se completa la post evaluación.

17. ¿Qué diferencia hay entre RAG y fine tuning?

RAG recupera conocimiento externo en cada consulta y es fácil de actualizar y citar. Fine tuning modifica el comportamiento del modelo mediante entrenamiento. Pueden combinarse y resuelven problemas distintos.

18. ¿Por qué una base vectorial puede devolver algo irrelevante?

Porque hace una búsqueda difusa y ordena vecinos por cercanía en el espacio del embedding, no por verdad. A diferencia de una consulta SQL exacta, siempre puede entregar el vector más cercano aunque sea malo. No existe un umbral universal y verificado que separe una respuesta válida de una inválida. En producción pueden configurarse umbrales y filtros evaluados, pero no garantizan corrección.

19. ¿Qué es una alucinación en un sistema RAG?

Como la base vectorial siempre responde y no hay un punto de corte universal, si recupera algo que no es lo que se necesita se lo llama alucinación. Técnicamente conviene separar dos fallas: recuperación irrelevante del índice y generación sin respaldo del LLM. Se reducen con mejores datos, filtros, reordenamiento, citas, abstención y evaluación.

20. ¿Qué diferencia hay entre agente y orquestador?

El agente realiza una capacidad acotada percibiendo y actuando. El orquestador decide la secuencia, condiciones y el intercambio entre agentes o herramientas. Los agentes por sí solos no saben nada: el valor depende de sus capacidades y de la coordinación. Un flujo rígido es orquestación. La planificación dinámica se acerca a colaboración.

21. ¿Qué aporta un sistema multiagente?

Especialización, modularidad y reemplazo de componentes con contratos claros. A cambio, aumenta la complejidad de coordinación, memoria, permisos, errores y observabilidad.

22. ¿Qué es amnesia de código?

Es perder comprensión del software al delegar generación y cambios sin lectura ni documentación. Se evita con arquitectura, módulos acotados, bitácoras, revisión, tests y trazabilidad.

23. ¿Por qué los patrones son útiles en vibe coding?

Expresan una estructura conocida con pocas palabras, delimitan responsabilidades y reducen el contexto que se debe enviar. También dan un marco para evaluar si el código generado es mantenible.

24. ¿Qué diferencia hay entre mantenimiento predictivo y prescriptivo?

El predictivo mira la bola de cristal: estima qué puede fallar y cuándo. El prescriptivo agrega la receta: recomienda qué acción tomar para evitar o mitigar la falla, considerando restricciones y consecuencias.

25. ¿Qué es un gemelo digital?

Es una representación digital conectada a un activo o proceso que reproduce estado y comportamiento. Permite simular, anticipar fallas y probar decisiones. Puede combinarse con PINN para imponer restricciones físicas conocidas. Su valor depende de calibración y actualización con el mundo real.

26. ¿Cuáles son los límites principales de la IA actual?

Aprende regularidades del pasado y funciona mejor en tareas tipificadas, con procedimiento conocido y variabilidad acotada. No comprende como una persona ni extrapola conocimiento entre dominios con la misma flexibilidad. El ejemplo de la camisa muestra que una tarea humana simple puede ser difícil cuando cada caso presenta una configuración nueva. La combinación buscada es velocidad y repetición de la IA, más criterio y abstracción de la inteligencia natural.

27. ¿Cuáles son las tres limitaciones del LLM y las tres ventajas de RAG?

LLM: fine tuning costoso, obsolescencia del conocimiento y opacidad de la respuesta. RAG: menor cómputo para incorporarlo, actualización rápida mediante documentos e índice, y trazabilidad hasta las fuentes recuperadas.

28. ¿Cómo se relaciona Dunning Kruger con los LLM?

El efecto describe la sobreconfianza que puede acompañar al conocimiento superficial. Los LLM suelen responder con certeza aunque la salida sea incompleta o incorrecta. El riesgo de adoptar esa seguridad como propia puede llamarse sesgo heredado. Se evita con conocimiento de base para interpelar, repreguntar, comparar y revisar.

29. ¿Por qué la métrica puede cambiar el algoritmo ganador?

Porque cada métrica representa un costo de error diferente. En el ejemplo numérico, un algoritmo tuvo recall de 85% y accuracy de 40%, mientras Gradient Boosting tuvo accuracy de 81% y recall de 35% sobre el mismo dataset. Si importan los falsos negativos gana el primero. Elegir accuracy hace ganar al segundo. La métrica debe definirse desde el problema, antes de ver el ranking.

30. ¿Por qué existen familias de algoritmos?

Porque antes de entrenar no se conocen por completo los patrones ni su distribución. Dos empresas pueden tener el mismo objetivo y columnas con nombres iguales, pero distinto volumen y distribución interna. Por eso no existe un mejor algoritmo universal: se selecciona una familia por el tipo de problema y se comparan sus algoritmos, tarea que AutoML automatiza en parte.

31. ¿Qué aporta el mecanismo de atención?

Permite ponderar qué tokens del contexto son relevantes para cada token procesado y calcular muchas relaciones en paralelo. Es la pieza central del Transformer y ayuda a mantener contexto amplio, pero no implica comprensión humana ni garantiza verdad.

32. ¿Qué necesita un prompt útil?

Claridad, detalle suficiente y contexto. El esquema ROC ordena rol, objetivo y contexto. En tareas complejas conviene dividir el trabajo, indicar formato y criterios de aceptación, revisar la salida y refinar la instrucción.
